# 深蓝色来源排查（未改代码，仅分析）

## 一、当前配置摘要

- **主题**（`frontend/ui.py` `launch_demo()`）：  
  `gr.themes.Soft(primary_hue="orange", neutral_hue="slate").set(block_background_fill="var(--card-bg)", input_background_fill="var(--input-bg)")`
- **未传 `secondary_hue`**，因此使用 Gradio Soft 的**默认次要色**。
- **我们的 CSS**（`assets/style.css`）：  
  对 `#ing-card-wrap`、`#page-step0 [class*="ing-card-wrap"]` 设了 `background: var(--shuai-cell-bg) !important`；对 `#ing-card-wrap > div` 设了 `background: transparent !important`；对 `#ing-card-group` 内 input/gr-box 等设了白底。

---

## 二、深蓝色最可能的来源

### 1. 未设置 `secondary_hue`，默认是蓝色（主因）

- Gradio 文档与主题说明：**Soft 主题的默认 secondary hue 是 blue**。
- 主题会从三个色相生成大量 CSS 变量：
  - **primary** → 主按钮、强调等；
  - **secondary** → 块背景、标签背景、边框、滑块、部分容器等；
  - **neutral** → 文字、中性背景等。
- 当前只传了 `primary_hue` 和 `neutral_hue`，**没有传 `secondary_hue`**，因此所有依赖「次要色」的变量都会用默认的**蓝色系**（如 *secondary_50、*secondary_100 等），这就是整块「深蓝/灰蓝」最可能的根源。

### 2. `.set()` 只改了两个变量，其余仍用主题默认

- 目前只覆盖了：
  - `block_background_fill="var(--card-bg)"`
  - `input_background_fill="var(--input-bg)"`
- 主题里通常还有大量相关变量，例如：
  - `block_label_background_fill`、`section_background_fill`、`container_background_fill`
  - 各种 `*_border_*`、`*_background_fill_*`
  - 滑块、下拉框、手风琴等组件的背景/边框
- 这些**没有被 .set() 覆盖**的变量，都会继续用主题默认值；在未改 secondary 的情况下，就会是**蓝色系**。所以即使「块背景」被我们改成米色，其它层或其它组件仍可能显示为蓝。

### 3. `.set()` 里使用 `var(--card-bg)` 可能未生效

- 若 Gradio 在生成主题 CSS 时，把 `var(--card-bg)` 当作**字面字符串**写入样式表，且该样式表在**独立作用域**（如 iframe、或早于我们 :root 的加载顺序）中解析，则：
  - 可能解析不到我们 `assets/style.css` 里在 `:root` 定义的 `--card-bg`；
  - 或主题引擎认为不是合法颜色而**回退到主题默认**（即 secondary 蓝）。
- 因此不能排除：**我们以为已经用 CSS 变量接管了块背景，但主题实际仍在使用默认的蓝色**。

### 4. DOM 层级：深蓝可能在「更内层」的 Gradio 容器上

- 我们只显式处理了：
  - `#ing-card-wrap`（米色）；
  - `#ing-card-wrap > div`（透明）。
- Gradio 在 Column 下可能有多层结构，例如：  
  `#ing-card-wrap` → 直接子 `div` → 再内层 `.block` / `.gr-form` / 其它 class。
- 若**深蓝背景是应用在再内一层的容器**上（例如某个带 `.block` 的 div），而我们只把「直接子 div」设为透明，那么：
  - 再内层仍使用主题的 `*secondary_*` 等变量；
  - 视觉上就会是「米色壳里还有一块深蓝」，或整块仍偏蓝。

### 5. 样式加载顺序与优先级

- `demo.launch(..., theme=custom_theme, css=_CSS, ...)` 中，主题和自定义 CSS 的**注入顺序**因 Gradio 版本可能不同。
- 若主题样式在 `_CSS` **之后**注入，且带有更高优先级的选择器或 `!important`，就会覆盖我们对 `#ing-card-wrap`、`#ing-card-wrap > div` 等的设置，导致我们看到的仍是主题色（蓝）。

### 6. 亮色/深色模式与 data-theme

- 已通过 `_HEAD_JS` 移除 `document.documentElement` 的 `dark` class 并用 MutationObserver 防止再次加上。
- 若 Gradio 还通过 **data-theme**、内联 style、或其它 class 应用「深色/蓝色」变量，仅移除 `dark` 可能不够，深蓝仍可能来自这些路径。

---

## 三、结论与建议方向（仍不改代码，仅建议）

| 可能原因 | 说明 |
|----------|------|
| **未设置 secondary_hue** | 默认 blue，所有 secondary 系变量（块、标签、边框等）都是蓝色系。 |
| **.set() 覆盖不全** | 只改了 2 个变量，其它块/容器/组件仍用默认（蓝）。 |
| **var() 在主题中未生效** | 主题可能回退到默认色（蓝）。 |
| **深蓝在更内层 DOM** | 我们只处理了 wrap 和其直接子 div，内层 .block 等仍用主题色。 |
| **主题 CSS 覆盖我们** | 加载顺序或优先级导致主题样式覆盖我们的米色/透明。 |

**建议的排查与修改方向（供后续实施）：**

1. **在 `launch_demo()` 中显式加上 `secondary_hue="orange"` 或 `secondary_hue="slate"`**，从根源上去掉默认蓝。
2. **扩大 `.set()` 覆盖范围**：至少加上与「块、容器、标签」相关的变量（如 `block_label_background_fill`、`section_background_fill` 等），值可用 `var(--card-bg)` 或与设计令牌一致的颜色。
3. **在浏览器开发者工具中**：对仍显示深蓝的区块「检查元素」，看它实际应用的 background 来自哪条规则、哪个 class、是否来自 theme 的 `*secondary_*` 或某条 Gradio 注入的 CSS，以确认是「未设 secondary」还是「内层容器/其它变量」导致。
4. **确认 Gradio 版本**：查当前版本的主题 API 是否支持在 `.set()` 中使用 `var(--...)`，以及主题 CSS 的注入顺序，便于决定是继续用 .set() 还是更多依赖外部 CSS 覆盖。

以上为仅分析、未改代码的排查结论；具体修改可在你确认后再做。

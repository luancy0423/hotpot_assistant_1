## 二、为什么还是深蓝色？

### 1. Gradio 的 DOM 不是「一层」

- `gr.Group(elem_id="ing-card-group")` 会对应一个带 `id="ing-card-group"`（或带数字后缀）的节点。
- Gradio 还会在这一层**外面**再包一层或多层 div（例如带 `.block`、`.gr-form` 等），用来做布局和主题。
- 主题会把这些**外层**容器的背景设为深色（如 `--block_background_fill` 等），所以：
  - **真正铺满整块、你看到的「整块底」往往是外层的那一层，不是 `#ing-card-group` 自己。**
- 若 `#ing-card-group` 对应的节点不是最外层，或者主题给外层写了更高优先级/`!important`，你只给 `#ing-card-group` 设 `background: var(--shuai-cell-bg)` 就会**被盖住**或**只生效在很小的区域**，看起来仍是深蓝。

### 2. 小结

- **逻辑**：我们只改了「Group 那一层」的背景，而**主题改的是「包住 Group 的那一层」的背景**，所以深蓝来自主题、米色没占上风。
- **结构**：不改 Python 的话，我们只能靠 CSS 猜「谁包住了 `#ing-card-group`」并一起设米色（例如用 `:has()` 选父级），或改主题/结构从根源上让「这一块」用米色。

## 三、两种修改方案（任选其一）

### 方案 A：纯 CSS——用 `:has()` 给「包裹卡片的父级」也设米色（不改 Python）

**思路**：不依赖 Gradio 内部类名（如 `.block`），只通过「谁包住了 `ing-card-group`」来选父级，给这些父级也设米色，并提高当前规则的优先级。

**修改位置**：只动 `assets/style.css`。

**具体做法**：

1. 给 `#ing-card-group`（及同组选择器）的 `background` 加上 `!important`，保证在主题下也能生效。
2. 增加两条规则，用 `:has()` 选中「直接父级」和「祖父级」（避免 Gradio 多包一层时仍生效）：
   - `#page-step0 div:has(> [id*="ing-card-group"])` → 直接包裹卡片的 div 设米色。
   - `#page-step0 div:has(> div > [id*="ing-card-group"])` → 再外一层的 div 也设米色。

**优点**：不改 UI 结构和 Python，只维护 CSS；不写死 `.block`，依赖的是「包含关系」。  
**缺点**：依赖 Gradio 的 DOM 结构（至少是「一层或两层 div 包住带 id 的节点」）；若 Gradio 升级改了结构，可能需再调选择器。

---

### 方案 B：改 UI 结构——用「自带米色类」的 Column 包一层（改 Python + CSS）

**思路**：在 Python 里给食材区加一层**我们完全可控**的容器（例如 `gr.Column`），并给它一个**仅用于样式的 class**（如 `ing-card-wrap`），在 CSS 里只认这个 class 设米色。这样不依赖 Gradio 内部 DOM，也不依赖 `elem_id` 落在哪一层。

**修改位置**：`frontend/ui.py` + `assets/style.css`。

**具体做法**：

1. **ui.py**：在 `gr.Group(elem_id="ing-card-group")` 外面再包一层 Column，并加上 `elem_classes`：
   ```python
   with gr.Column(elem_id="ing-card-wrap", elem_classes=["ing-card-wrap"]):
       with gr.Group(elem_id="ing-card-group"):
           # 原有内容不变
   ```
   （若希望避免多一层布局影响，可仅用 `elem_classes=["ing-card-wrap"]`，不用 `elem_id` 也可。）

2. **style.css**：新增只针对该容器的规则，例如：
   ```css
   #page-step0 .ing-card-wrap,
   #page-step0 [class*="ing-card-wrap"] {
     background: var(--shuai-cell-bg) !important;
     border-radius: var(--card-radius);
     padding: var(--card-padding);
     margin: var(--card-margin);
     box-shadow: var(--shuai-shadow);
   }
   ```
   保留原有 `#ing-card-group` 的样式（可去掉其 background，让外层 .ing-card-wrap 负责底色，避免重复）。

**优点**：不依赖 Gradio 内部结构，我们控制的「壳」一定存在；以后要改米色或加边框等只改这一处即可。  
**缺点**：需要改一处 Python 结构并多维护一个 class；多一层 DOM。

---

## 四、对比与建议

| 维度           | 方案 A（纯 CSS + :has()）     | 方案 B（Python 加一层 + class） |
|----------------|-------------------------------|----------------------------------|
| 改动范围       | 仅 `assets/style.css`         | `frontend/ui.py` + `assets/style.css` |
| 对 Gradio 依赖 | 依赖其当前 DOM 层级           | 不依赖，自己加一层壳            |
| 可维护性      | 结构一变可能要调选择器       | 结构清晰，样式挂钩明确          |
| 是否写死 .block | 否，用父子关系选             | 否，用自有 class                 |

- **想最少动代码、先快速解决**：选 **方案 A**。  
- **希望长期稳定、不跟 Gradio 内部 DOM 绑定**：选 **方案 B**。

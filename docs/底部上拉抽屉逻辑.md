# 底部上拉抽屉 · 现有逻辑梳理

本文档梳理「底部购物车栏 + 上拉抽屉」的现有实现逻辑；「已添加的食材」仅保留页面内常驻表格，不再使用上拉抽屉。

---

## 正确触发方式与排查（点击没反应时）

1. **脚本挂到哪**  
   事件会绑在「实际包含页面的 document」上：
   - 当前页的 `document`；
   - 当前页里所有 **iframe** 的 `contentDocument`（Gradio 若嵌在 iframe 里，点击在 iframe 内发生）。
   绑定会在 100ms、500ms、1500ms 各尝试一次，避免应用晚挂载。

2. **正确触发方式**  
   - 必须进入 **第一步 · 输入食材**（点「开始定制」后再点一次「下一步」或从首页进入步骤 1）。  
   - 底部栏为 **纯 HTML**（无单独「打开购物车」Gradio 按钮）：点击底部栏左侧 **`.bsk-left` 区域**（🛒 图标或预览文案）即可打开上拉抽屉，与参考的「美团/饿了么」底部栏一致。

3. **如何自查「有没有绑对」**  
   - 打开浏览器开发者工具 (F12) → Console。  
   - 若应用在 **主页面**：输入  
     `document.querySelector('.shuai-basket-area')`  
     有返回值说明底栏在当前 document，我们的监听应能收到点击。  
   - 若应用在 **iframe**：先切到该 iframe 的 context（Console 顶部的「top」下拉里选对应 iframe），再在同一 iframe 下执行上面同一句；有返回值说明底栏在该 iframe 的 document，我们已对该 iframe 的 document 做了绑定。  
   - 若点击后抽屉能正常上拉，说明事件委托已生效；若 **完全无反应**，多半是应用跑在另一个 iframe 或另一层 document，当前脚本没挂到那里。

4. **手动测打开（确认脚本是否执行）**  
   在 Console 里执行（在包含 `.gradio-container` 的那个 window 下）：  
   `document.querySelector('.bsk-left').click()`  
   若抽屉能打开，说明该 document 上已绑定、且选择器正确，问题在真实用户点击没进这个 document。

## 触发条件速查

| 动作 | 触发条件（JS 判断） | 前置条件 |
|------|---------------------|----------|
| **打开底部购物车抽屉** | 点击目标为 `.bsk-left` 或其子节点（如 `.bsk-icon`、`.bsk-preview`）；`e.target` 或 `e.composedPath()` 中某节点满足 `node.closest('.bsk-left')` 且能取到 `.shuai-basket-area` | 当前为「第一步 · 输入食材」页（`step0` 可见），底栏在 DOM 内且可点击 |
| **关闭底部购物车抽屉** | 点击 `.shuai-overlay`（遮罩）或 `.shuai-drawer-close`（✕）或 `.shuai-drawer-next`（抽屉内「下一步」） | 抽屉已打开（overlay/drawer 已在 body 上并显示） |

- **事件绑定**：页面 load 后约 100ms 在 `document` 上注册**捕获阶段**点击监听（`addEventListener(..., true)`），故任意子元素点击都会先经过该 handler。
- **Shadow DOM / 跨边界**：若点击发生在 Shadow Root 内，`e.target` 可能为 host，已用 `e.composedPath()` 遍历路径再次解析 `.bsk-left`，避免漏触发。

## 二、底部购物车栏 + 上拉抽屉（购物车）

### 2.1 在页面中的位置

- **仅出现在「第一步 · 输入食材」页**：底部栏在 `step0`（`page-step0`）内，与「已添加的食材」表格、删除行、下一步等同一 Column。
- 结构：`step0` → … → `gr.Row(elem_id="basket-bar-row")` → **左侧** `gr.HTML(basket_bar_html(0, []))`（整条购物车栏 + 遮罩 + 抽屉均为该 HTML），**右侧** `gr.Button("下一步 ›", elem_id="btn-next-in-bar")`。抽屉内「下一步」由 JS 触发该按钮，与参考的「隐藏按钮 + 前端触发」模式一致。
- `#basket-bar-row` 用 CSS 固定到底部：`position: fixed; bottom: 0; left: 50%; transform: translateX(-50%); width: 450px; z-index: 50`。

### 2.2 HTML 结构（由 `basket_bar_html(count, state)` 生成）

```
.shuai-basket-area
├── .shuai-basket-bar           ← 用户看到的「底栏」
│   ├── .bsk-left               ← 可点击：打开抽屉（🛒 + 预览文案 + badge）
│   └── .bsk-right              ← 「共 N 件涮品」
├── .shuai-overlay              ← 遮罩，初始 style="display:none"
└── .shuai-drawer               ← 抽屉面板，初始在视口下方（translateY(100%)）
    ├── .shuai-drawer-handle
    ├── .shuai-drawer-header    （标题 + ✕）
    ├── .shuai-drawer-body      ← basket_drawer_html(state)，商品列表
    └── .shuai-drawer-footer    （「下一步」按钮）
```

- `count` / `state` 变化时，整段 HTML 会由 Gradio 更新（见 2.5），因此 overlay/drawer 的 `id` 里带 `count`（如 `shuai-overlay-0`）。
- HTML 里还带内联 `<script>`，会定义 `shuaiOpenBasket` 等；**Gradio 不执行该 script**，实际生效的是 `_DRAWER_JS` 里的事件委托。

### 2.3 打开 / 关闭的交互逻辑（由 `_DRAWER_JS` 实现）

- **绑定时机**：`demo.load(js=_DRAWER_JS)`，脚本内 `setTimeout(attach, 100)` 在约 100ms 后往 `document` 上挂一次捕获阶段点击监听。
- **打开**：点击目标在 `.bsk-left` 内时：
  1. `area = target.closest('.shuai-basket-area')`
  2. `o = area.querySelector('.shuai-overlay')`，`d = area.querySelector('.shuai-drawer')`
  3. 把 `o`、`d` 依次 `appendChild` 到 `document.body`（避免被父级 transform/overflow 裁剪）
  4. 对 overlay 设 `display='block'`，并内联 `zIndex/position/inset`；对 drawer 设 `zIndex/position` 并 `classList.add('open')`
  5. 抽屉通过 CSS `.shuai-drawer.open { transform: translateY(0) }` 做上拉动画
- **关闭**：
  - 点击 `.shuai-overlay`：用 overlay 上存的 `_area`、`_drawer`，先隐藏、去掉 `open`，再把 overlay 和 drawer 移回 `area`。
  - 点击 `.shuai-drawer-close`：通过 `drawer.previousElementSibling` 找到当前在 body 下的 overlay，再执行同样的关闭逻辑。
  - 点击 `.shuai-drawer-next`：先执行上述关闭逻辑，再 `setTimeout` 约 120ms 后触发 `#btn-next-in-bar` 里的按钮 click（跳下一步）。

### 2.4 样式要点（ui.py 内 _CSS）

- 遮罩：`.shuai-overlay` — `position: fixed; inset: 0; z-index: 99998 !important`，打开时再内联保证在最前。
- 抽屉：`.shuai-drawer` — 默认 `transform: translateX(-50%) translateY(100%)`；`.shuai-drawer.open` — `translateY(0)`；`z-index: 99999 !important`；`transition` 做平滑上拉。

### 2.5 数据流：何时更新底部栏与食材表格

以下操作会更新 `ingredient_table_state`，并同时：
- 把 `basket_bar_html(len(state), state)` 写入 `basket_bar_html_comp`（刷新底部栏 + 抽屉内列表）
- 把 `ingredient_table_html(state)` 写入 `ingredient_table`（刷新页面内「已添加的食材」表格）

涉及事件：加入清单、删除所选行、批量添加、识图填入、识别语音、商家系统占位。

## 三、「已添加的食材」页面内展示（无抽屉）

- 在 **第一步 · 输入食材** 页（`step0`）中，位于商家按钮之下、删除行之上。
- 结构：`gr.HTML('🧺 已添加的食材')` → `gr.HTML(value=ingredient_table_html([]), elem_id="ingredient-table-html")` → 删除行（下拉 + 删除所选行按钮）。
- 表格常驻展示，由 `ingredient_table_html(state)` 渲染；与底部购物车抽屉内的 `basket_drawer_html(state)` 数据一致、展示形式不同（表格 vs 列表）。

## 四、可能的问题点（便于你排查）

1. **点击无反应**  
   - 事件绑定在 `document`，且延迟 100ms；若 Gradio 在 iframe 或 Shadow DOM 中渲染，实际点击可能发生在子 document，当前脚本若运行在父 document 则收不到点击。
   - 可尝试：在控制台确认点击时 `e.target.closest('.bsk-left')` 是否非 null，以及 `document.querySelector('.shuai-basket-area')` 是否存在。

2. **打开后看不见**  
   - 已通过把 overlay/drawer 移到 `document.body` 并设高 z-index（99998/99999）规避父级裁剪/遮挡。
   - 若仍被挡：检查是否有其他全局元素 z-index 更大，或父级 `isolation/transform` 导致 fixed 相对错误容器。

3. **内联 script 无效**  
   - 设计上不依赖 HTML 里的 `<script>`；所有行为以 `_DRAWER_JS` 的委托为准。若删除内联 script，逻辑应不变。

4. **底部栏只在第一步出现**  
   - 由布局决定：`#basket-bar-row` 在 `step0` 内，仅当「第一步 · 输入食材」页可见时显示；首页无该栏属预期。
